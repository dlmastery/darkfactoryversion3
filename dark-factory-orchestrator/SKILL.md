# Dark Factory Orchestrator

## Purpose

Coordinate a full dark-factory run while keeping context lean. Select the right child skills, create the lifecycle plan, enforce evidence gates, and keep humans able to take over or hand work back at any stage.

## Start Here

1. Decide whether this is a tiny non-factory task or a governed DFMS run.
2. If governed, require `df-meta-attractor` first and attach an Attractor Run Record, or record an explicit waiver with owner, rationale, expiry, and residual risk.
3. If governed, require an approved engagement governance record from `df-governance-mayor` or explicit waiver before material token spend: client owner, delivery owner, rough token SWAG, iteration checkpoint, change-control policy, and reapproval trigger.
4. If governed, require a knowledge graph seed and a strict `TASKS.md` bead ledger, or record an explicit waiver with owner, rationale, expiry, and residual risk.
5. If governed, require a TPM flow ledger and PERT dependency plan; the active/next work must be selected from those records.
6. If governed, require an AI judge/jury transition record for material state movement, unless the transition is a boundary-human approval item.
7. If governed, run the execution-kernel preflight from `df-governance-mayor` and obey its `legal_next_action`; if it says `closed`, open a new bead/change request before continuing.
8. If governed, require an SDLC Stage Coverage Matrix from `df-governance-mayor`; do not continue when a required stage or test class is missing without waiver.
9. If this run was produced by a meta-meta/generator path, require a Product Tailoring Profile and Generated Meta-Skill Contract; refuse generic generated factories.
10. Require a passing Hawkeye Conformance Audit Record for governed continuation, stage exit, artifact pass, generated meta-skill handoff, certificate, or closure.
11. For governed project-book, code, artifact, or meta-skill outputs, require a Human Review and Onboarding Portal before handoff, certificate, or closure, or record an explicit waiver with owner, rationale, expiry, and residual risk.
12. Classify the request as greenfield, brownfield, artifact-only, review-only, handoff, recovery, portal/update, or governance update.
13. Read `references/routing.md` if the mode is unclear.
14. Load only the child skill needed for the current node.
15. Create a node contract with `what`, `why`, `how`, `where`, `when`, `who`, and `how_good`.
16. Create or update the current task bead before material execution.
17. Require standards mapping, expert debate, traceability, verification evidence, Hawkeye conformance evidence, human review portal evidence when applicable, and handoff notes for material outputs.

## Operating Rules

- Do not claim zero mistakes. Build a zero-defect posture through redundant review, evidence, and residual-risk records.
- Do not run a governed DFMS task without an Attractor Run Record or explicit waiver.
- Do not start material governed work without approved engagement governance: scope baseline, rough token SWAG, iteration checkpoint, and change-control policy.
- Do not start material governed work without a task bead in `TASKS.md` and a corresponding knowledge graph node.
- Do not start material governed work without a TPM flow step, PERT dependency node, and valid ready/active transition.
- Do not accept material governed work without an AI judge/jury verdict or explicit boundary-human approval.
- Do not accept app/product/code work as complete with documents only. Code-producing work requires implementation and test evidence; UI work requires browser/WYSIWYG evidence; scenario-driven work requires scenario, holdout, and transfer evidence; production work requires release/ops evidence.
- Do not accept governed project-book, artifact, code, or meta-skill work as handoff-ready unless a human review/onboarding portal indexes the current state, all top-level Markdown docs, structured records, evidence, Mermaid diagrams, stage gates, decisions, reviews, certificates, and next safe action, or an explicit waiver is recorded.
- Do not continue an iteration that exceeds its token SWAG or changes scope without client reapproval.
- Do not expand a governed project spec into PRD/SRS/design/implementation without an accepted interrogation record, recursive spec decomposition record, intake package validation, or explicit owner-scoped waiver.
- Do not edit a brownfield codebase before recon and impact analysis.
- Do not accept an artifact that lacks a traceable source, reviewer scores, and residual-risk state.
- Do not accept governed work that lacks control graph, work ledger, and refinery gate proof, unless each missing object has an explicit waiver.
- Do not accept a bead that lacks trace links, gate result, evidence, and next bead or closure rationale.
- Do not let methodology labels become decoration. If ISO, RUP, MDA, DDD, TDD, BDD, SRE, or SSDF is invoked, map it to concrete artifacts and checks.
- Keep the user-facing conversation simple. Store technical evidence in files.

## Default Gates

- Attractor gate: Attractor Run Record or explicit waiver.
- Control graph gate: material work maps to a control graph node.
- Work ledger gate: material work has owner, status, evidence, and next action.
- Knowledge graph gate: material objects are typed nodes with resolving typed edges, no orphan accepted claims, and no template-only proof.
- Task bead gate: material work appears in `TASKS.md` with state, owner, control node, work ledger, KG node, gate, evidence, token posture, approval, and next bead.
- TPM flow gate: material work appears in the TPM flow ledger with state, owner, dependencies, entry/exit criteria, judge/jury record, evidence, and next step.
- PERT gate: every active or accepted step has resolved predecessors, no dependency cycles, critical path visibility, and legal next-step candidates.
- AI judge/jury gate: TPM Judge, Evidence Clerk, specialist jurors, adversarial prosecutor, and foreperson verdict approve material transitions inside approved human boundaries.
- Execution-kernel gate: `TASKS.md`, TPM flow, PERT, knowledge graph, and judge/jury records are cross-validated before any next action is chosen.
- SDLC stage coverage gate: every stage is accepted, waived, deferred, or not applicable with rationale; required testing classes are evidenced; docs-only completion is blocked for product/code/UI/production claims.
- Meta-meta compiler gate: generated factories must be product-tailored with archetype, surfaces, risks, method blend, artifacts, testing, runtime gates, ledgers, refusal rules, and validation records.
- Hawkeye conformance gate: every required stage and process is audited; Hawkeye veto blocks continuation, pass, certificate, or closure.
- Human review portal gate: onboarding dashboard, machine-readable portal index, diagram atlas, artifact/evidence coverage, role-based reading paths, and validation evidence are present before handoff or closure for governed project-book work.
- Engagement governance gate: client/dark-factory owners, token budget SWAG, iteration plan, and change-control approvals are recorded.
- Refinery gate: acceptance or rework is recorded before final.
- Governance compliance: 98 percent.
- Selected expert review: 96 percent per expert.
- Mandatory traceability: 100 percent or explicit deferral.
- Critical and high security findings: zero unresolved.
- Production readiness: owner, deploy, rollback, observability, incident path, and known risks all present.
