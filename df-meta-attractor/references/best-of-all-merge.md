# Best-Of-All Merge

Use this when the user asks to merge the best patterns from external dark-factory, software-factory, agent, workflow, or DevOps systems.

## Principle

DFMS remains the standards/governance spine. External systems contribute optional patterns:

- StrongDM Attractor: NLSpec-first implementability, graph discipline, human gates.
- Fabro: executable workflow graphs, model routing, checkpoints, verification, sandboxes, API/UI, retrospectives.
- Gas Town: durable work state, coordinator/worker model, health monitoring, escalation, merge refinery, predecessor recovery.
- Octopus Garden AI: discovery, rollout, training, support, business process fit.
- Fabriqa.ai: PRD-to-code traceability and validation-layer framing.
- Factory.ai: delegation surfaces where humans already work.
- Octopus Deploy: deployment, troubleshooting, remediation, audit, operations lifecycle.

## Required Merged Objects

- Control graph: lifecycle nodes, child skills, human gates, evidence outputs, and re-entry paths.
- Work ledger: durable work items for requirements, tasks, artifacts, risks, decisions, tests, handoffs, evidence, escalations, and retrospectives.
- Refinery gate: acceptance gate combining tests, traceability, expert rubrics, security/ops checks, and residual risk.
- Runtime adapter boundary: optional execution adapters must not bypass DFMS governance.

## Acceptance Rule

Adopt an external pattern only when it:

- maps to user requirements or DFMS principles,
- strengthens standards, traceability, evidence, handoff, or verification,
- works for both greenfield and brownfield use,
- has a clear owner and verification path,
- does not make a vendor or runtime mandatory.
